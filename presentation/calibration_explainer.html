Architecture

The two-pool simulator

The simulator runs two CPMM pools side-by-side. A router optimally splits each retail order between them. The submission pool (5bp fee) represents the target pool; the normalizer represents aggregate competing liquidity.

Calibration

Simulation-based binary search

We calibrate the normalizer depth by running the full simulator at each candidate value and measuring the resulting volume share. A binary search converges on the depth that matches the on-chain target.

1

Set the 5bp pool's virtual USDC reserve to match on-chain data from the real Uniswap v3 WETH/USDC 5bp pool. Set normalizer fee to 30bps.

2

Binary search over depth

For each candidate normalizer depth, run the full sim (regime-switching prices, empirical orders, arb agents). Measure the resulting volume share on the submission pool.

lo, hi = $10M, $500M
while hi − lo > ε:
  mid = (lo + hi) / 2
  vol_share = run_sim(mid)
  if vol_share > 63.4%:
    lo = mid ← need more competition
  else:
    hi = mid
3

Converge on target

The search converges when the simulated volume share matches the on-chain target of 63.4% (from pool_stats_6m.csv). This is the one free parameter.

Why Simulation?

The analytical formula doesn't work in practice

The routing formula f = Ls√γs / (Ls√γs + Ln√γn) is a first-order approximation for a single trade in static pools. The actual simulation dynamics diverge because of:

⚡

Arbitrageurs

Arb agents realign pool prices after every block. This changes the effective liquidity each pool offers to the next trade, creating feedback loops the static formula can't capture.

📈

Price movements

External price follows a regime-switching process. Large moves create asymmetric state across pools — one pool is stale, affecting routing splits in ways that compound over time.

🔄

State-dependent routing

Each trade changes pool reserves. The next trade's optimal split depends on all prior trades. Over thousands of blocks, these path dependencies accumulate significantly.

Fee barely matters — depth dominates routing

Evaluation

Validate via LP markout distribution

Why markouts, not volume share?

Volume share was the calibration target — it's "used up" and can't also serve as validation. Instead, we compare the LP markout distribution (LP profit per trade, in bps) from the calibrated simulator against observed on-chain markouts. If the sim produces realistic markouts without being tuned to them, that's genuine out-of-sample validation.

Bottom Line

Calibration uses simulation search, not analytical formulas. We binary-search over normalizer depth, running the full simulator at each step, until volume share matches the on-chain target of 63.4%. Validation uses an independent metric — LP markout distributions — compared against observed on-chain data from Uniswap's markout tables. The analytical formula is a useful first approximation but the sim search is necessary because arb agents, price dynamics, and state-dependent routing make the actual dynamics diverge.
